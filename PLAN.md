### Step 2: `tokenwise audit` — Scanner

Build the file discovery engine.

**2a. File scanner** (`src/audit/scanner.ts`)
- Find instruction files in priority order:
  1. `CLAUDE.md` in project root
  2. `AGENTS.md` in project root (takes priority over CLAUDE.md per OpenCode spec)
  3. `.claude/rules/*.md` (Claude Code rules)
  4. `.opencode/agents/*.md` (OpenCode agent definitions)
  5. `~/.claude/CLAUDE.md` (global Claude instructions)
  6. `~/.config/opencode/AGENTS.md` (global OpenCode instructions)
- For each file: path, exists, token count, line count
- Return `ScanResult[]` with file metadata

**2b. Rules frontmatter parser** (`src/audit/scanner.ts`)
- Parse YAML frontmatter from `.claude/rules/*.md`
- Detect `paths:` field presence
- Classify: `scoped` (has paths) vs `unscoped` (loads always)
- Count tokens for scoped vs unscoped rules separately

**Edge cases:**
- No instruction files found → return empty array with helpful message
- Symlinked files → follow, don't recurse
- Binary files → skip
- Permission denied → log, continue

**Tests:**

### Step 3: `tokenwise audit` — Heuristics

Build the audit flag engine.

**3a. Heuristic rules** (`src/audit/heuristics.ts`)

Each heuristic is a pure function: `(content: string, filePath: string, context: AuditContext) => Flag[]`

| Heuristic ID | What it detects | Threshold | Example |
|---|---|---|---|
| `excessive_length` | File over 200 lines | >200 lines | 867-line CLAUDE.md = instant flag |
| `high_boilerplate` | Generic AI advice Claude already knows | >25% of non-header lines match boilerplate regex | "Always write clean code" |
| `excessive_emphasis` | ALL-CAPS emphasis (NEVER, ALWAYS, MUST, CRITICAL) | >3 lines with shouty words | "NEVER use var" repeated 10x |
| `duplicate_section` | Sections with >60% jaccard similarity | jaccard >0.6 between any two sections | "Code Style" and "Formatting" sections saying the same thing |
| `unscoped_rule` | `.claude/rules/` file without `paths:` frontmatter | Missing `paths:` key | A rule about database patterns that loads during frontend work |
| `redundant_with_config` | Instruction duplicated in linter/formatter config | Instruction text matches config value | "Use 2-space indent" when .prettierrc has tabWidth: 2 |
| `obvious_description` | Describes directory structure that's self-evident | Regex match on obvious patterns | "`src/` contains source code" |
| `empty_section` | Section header with no content | Section body is only whitespace | `## Architecture` followed by `## Testing` |

**Boilerplate regex patterns:**
```typescript
const BOILERPLATE = [
  /always\s+(write|use|follow|ensure|make)\s+(clean|maintainable|readable|best|proper|good)/i,
  /follow\s+(best\s+)?practices/i,
  /use\s+(proper|appropriate|correct)\s+(naming|coding|formatting)/i,
  /make\s+sure\s+(your\s+)?code\s+is/i,
  /prefer\s+(clean|simple|readable)\s+code/i,
  /write\s+(well-structured|well-organized|well-documented)\s+code/i,
];
```

**Section extraction:**
- Split markdown on `^#{1,3}\s+` headers
- For each section: extract title, body, compute jaccard similarity against all other sections
- Shingling: k=5 word n-grams, jaccard on set intersection/union

**3b. Audit context** (`src/audit/heuristics.ts`)
- `AuditContext`: project root path, list of config files found (.eslintrc, .prettierrc, tsconfig.json), their parsed values
- Cross-file comparison: read config files, compare CLAUDE.md instructions against config values

**3c. Aggregate audit result** (`src/audit/types.ts`)
```typescript
interface AuditFlag {
  id: string;           // heuristic ID
  severity: "high" | "medium" | "low";
  message: string;      // human-readable description
  line?: number;        // line number if applicable
  savings?: number;     // estimated token savings from fixing this
}

interface FileAuditResult {
  path: string;
  tokens: number;           // total tokens in file
  lines: number;
  flags: AuditFlag[];
  reducibleTokens: number;  // sum of flag.savings
  loadFrequency: "every_message" | "conditional" | "on_demand";
}

interface AuditResult {
  files: FileAuditResult[];
  totalTokens: number;
  reducibleTokens: number;
  reduciblePercent: number;
  monthlyCostEstimate: number;     // at Sonnet pricing, 50 msg/day
  monthlySavingsEstimate: number;  // after reducing
}
```

**Tests:**
- `tests/audit/heuristics.test.ts` — each heuristic against sample content (bloated CLAUDE.md, minimal CLAUDE.md, rules with/without paths, duplicate sections, boilerplate-heavy content)

**Verification:** `npx vitest run` all tests pass.

### Step 4: `tokenwise audit` — Reporter & Fixer

Build the output and fix system.

**4a. Audit reporter** (`src/audit/reporter.ts`)
- Print per-file table: Path | Tokens | Lines | Load Frequency | Flags | Savings
- Print summary: total tokens, reducible tokens, % reducible
- Print monthly estimate: "At Sonnet pricing (50 messages/day), your instruction files cost ~$X/month. After optimization: ~$Y/month. Savings: ~$Z/month."
- Color coding: red=high severity, yellow=medium, green=savings
- `--json` mode: dump `AuditResult` as JSON

**4b. Audit fixer** (`src/audit/fixer.ts`)
- `--fix` flag: generates trimmed versions
- Safe transforms only:
  1. De-boilerplate: remove lines matching boilerplate regex
  2. De-shoutify: convert "NEVER" → "Never", "ALWAYS" → "Always", "MUST" → "Should"
  3. Dedupe sections: merge sections with jaccard >0.6 (keep the first, drop the second)
  4. Remove empty sections
  5. Add `paths:` frontmatter stub to unscoped rules (comment: "# TODO: add file paths to scope this rule")
- Before writing: create `.bak` backup
- After writing: print diff, new token count, estimated savings
- Never auto-delete content. Only reorganize and soften.

**Tests:**
- `tests/audit/fixer.test.ts` — each transform against sample content, verify backup creation, verify token reduction

**Verification:** `npx vitest run`, then manual test: `npx tokenwise audit --dir ~/gstack` (uses the 867-line CLAUDE.md found on this machine as a real test case).

### Step 5: `tokenwise scan` — Claude Code parser

Build the Claude Code session log analyzer.

**5a. JSONL discovery** (`src/scan/claude-code.ts`)
- Search `~/.claude/projects/` recursively for `*.jsonl` files
- Sort by modification time, select latest session by default
- `--session <path>` flag to override

**5b. JSONL parser** (`src/scan/claude-code.ts`)
- Read JSONL line by line
- Filter for `type: "assistant"` messages (these carry `message.usage`)
- Extract per-message: `message.usage.input_tokens`, `output_tokens`, `cache_creation_input_tokens`, `cache_read_input_tokens`
- Track message sequence number for temporal analysis
- Compute: total input, total output, total cache_read, total cache_creation

**5c. Token breakdown** (`src/scan/claude-code.ts`)
- **System prompt estimate**: tokens in the first user message (the initial context load)
- **History/context**: cumulated input tokens minus first message minus latest tool calls
- **Tool output**: sum of `tool_result` type messages' token estimates (using counter)
- **Code reads**: sum of `tool_use` messages where tool is Read/Grep/Glob, estimate tokens from content
- **Cache hit rate**: `cache_read_input_tokens / (input_tokens + cache_read_input_tokens)`

**Edge cases:**
- Malformed JSONL line → skip, increment warning counter
- Session with zero messages → report empty
- Multiple sessions in same project → use latest by mtime
- Very large JSONL files (100MB+) → stream, don't load into memory

**Tests:**
- `tests/scan/claude-code.test.ts` — parse sample JSONL with known token counts, verify breakdown accuracy

**Verification:** `npx vitest run`, then `npx tokenwise scan` against a real Claude Code session on this machine.

### Step 6: `tokenwise scan` — OpenCode parser

Build the OpenCode SQLite session analyzer.

**6a. SQLite discovery** (`src/scan/opencode.ts`)
- Search for `opencode.db` at standard paths:
  - `~/.local/share/opencode/opencode.db`
  - `%APPDATA%/opencode/opencode.db` (Windows)
  - `XDG_DATA_HOME/opencode/opencode.db` (Linux)
- `--db <path>` flag to override
- Open in read-only mode, WAL journal

**6b. SQLite parser** (`src/scan/opencode.ts`)
- Query `session` table: get latest session by `time_created`
- Query `message` table for that session: extract `data->tokens` JSON fields
- Per-message: `input`, `output`, `cache.read`, `cache.write`, `reasoning`
- Aggregate: total input, output, cache_read, cache_write, reasoning, cost

**6c. Token breakdown** (`src/scan/opencode.ts`)
- **System prompt**: tokens from first assistant message in session
- **Tool output**: query `part` table for `type='tool'` parts, estimate token cost of tool inputs/outputs
- **Cache hit rate**: `cache_read / (input + cache_read)`
- **Reasoning overhead**: `reasoning / total_input` ratio

**Edge cases:**
- SQLite locked → use read-only mode with WAL
- Empty database → report no sessions
- Multiple sessions → use latest by `time_created`
- `--session <id>` to select specific session

**Tests:**
- `tests/scan/opencode.test.ts` — create temp SQLite with known data, verify queries
- Integration test: real `opencode.db` on this machine has 24 sessions with 8.4M input tokens

**Verification:** `npx vitest run`, then `npx tokenwise scan` against the local OpenCode database.

### Step 8: CLI integration

Wire everything together.

**8a. CLI entry** (`src/cli.ts`)
```
tokenwise audit [--dir <path>] [--fix] [--json] [--model <model>] [--verbose]
tokenwise scan [--session <path>] [--db <path>] [--json] [--model <model>] [--verbose]
tokenwise --help
tokenwise --version
```

**8b. Help text**
```
tokenwise — find out where your AI coding agent burns tokens

Usage:
  tokenwise audit    Analyze instruction files (CLAUDE.md, AGENTS.md, rules/)
  tokenwise scan     Analyze session logs (Claude Code + OpenCode)

Options:
  --dir <path>       Project directory (default: cwd)
  --fix              Auto-fix issues found by audit (writes .bak backups)
  --session <path>   Path to specific session file/database
  --db <path>        Path to OpenCode database
  --model <model>    Model for cost estimation: sonnet (default), haiku, opus
  --json             Output raw JSON
  --verbose          Show per-flag details

Examples:
  tokenwise audit              # Scan current project's instruction files
  tokenwise audit --fix        # Scan and auto-fix
  tokenwise scan               # Analyze latest Claude Code session
  tokenwise scan --model opus  # Estimate at Opus pricing
```

**8c. Build & bin**
- `tsconfig.json` outDir: `dist/`
- `package.json` bin: `"tokenwise": "./dist/cli.js"`
- Shebang: `#!/usr/bin/env node`
- Build: `npx tsc && chmod +x dist/cli.js`
